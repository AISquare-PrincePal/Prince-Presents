Product Requirements Document · V1

Turning Team OS into the app your whole team opens every morning.

Team OS is the AISquare team's AI-run cockpit: one place to start the day, see what is on track, chase what is owed, and let Claude do the heavy lifting (drafting, planning, summarizing, chasing). It works today, but only for one power user in a terminal. This plan turns it into something every teammate in sales, marketing, design and GTM can open and use without ever touching a command line.

The shift A developer tool you run in a terminal → A desktop app your whole team double-clicks
1 Product definition

What Team OS is, why it exists, and where it goes

The PRD framing: a formal definition and one-line summary, the problem it solves, and the long-term vision that V1 is the first usable slice of.

One-line summary. Team OS is a local desktop app that gives the whole AISquare team a single, AI-run cockpit for their work: a calendar, a capacity board, an accountability view, customer and roadmap surfaces, and a command runner. An AI engine runs locally in the background, reads the team's shared knowledge, does the heavy lifting (drafting, planning, summarizing, chasing), and the app is the human surface where people see it and act on it.
i Definition

Team OS is the team-facing application layer of AISquare's compounding-memory system — a local desktop cockpit plus a local AI engine that turns the shared, domain-organized knowledge base into daily action for every role, not just the person who built it.

Why this exists · the problem

A forward-deployed product team runs on scattered context. Customer calls live in transcripts, commitments live in inboxes, plans live in someone's head, competitor moves live in a newsletter, the roadmap drifts from what customers actually said, and "who is free and by when" takes a day to answer. Knowledge does not compound. The same questions get re-answered every week.

We have already solved the brain: a structured knowledge base plus a library of AI commands and skills that capture calls, triage email, plan sprints, track accountability, and keep the roadmap honest. What it lacks is a face. Today it runs through a chat and terminal — fine for one power user, a non-starter for a team. Team OS is that face.

Vision · where this goes

The long-term vision is a shared operating system for the whole team where:

  • Every customer signal, decision, and commitment is captured once and reused everywhere.
  • The roadmap updates itself from real customer feedback rather than drifting.
  • Anyone can ask "what is on track, who is free, what is at risk" and get an answer in seconds.
  • Each person has an AI chief of staff that runs their daily rituals and holds the team accountable.
  • Knowledge graduates from private notes into a shared source of record, with a hard wall around what stays personal.

V1 is the first usable slice: a local app for the team, focused on the daily rituals and the views that answer the most expensive recurring questions.

⚙ How the app is built and accessed
  • Native desktop app (Tauri). Built as a Tauri app so it has native system access while running fully local on each teammate's machine — no cloud dependency to open and use it.
  • Design-system-driven UI. Screens are built from a shared component library backed by a Figma library, so designs can be handed to and parsed by an agent to generate and maintain screens consistently.
  • Local now, hosted later. V1 is a local app for the team. A future plan (owned by Gungjan) hosts Team OS on a domain so external contractors can access it too.
3 Honest snapshot

Where we are today

A strong foundation is already shipped. The value is real and built. The barrier is entirely the surface.

Strong foundation, shipped

  • A working desktop UI: chat-first cockpit, board, calendar, roadmap, customers, files, personal ledgers.
  • Review-then-approve cards, a first-run setup wizard, and a privacy wall for personal data.
  • The whole brain: domain knowledge plus roughly 25 AI commands and skills.

But still a power-user tool

  • The engine is the claude command-line binary. The app launches it in the background.

4 Why this matters now

The brain compounds value every week, but only one person can reach it

A forward-deployed team runs on scattered context: calls, inboxes, Slack, someone's head. Team OS already solves that, for whoever is comfortable in a terminal. Everyone else is locked out. Five concrete barriers:

1

You must be a developer to even start

Node, git clone, terminal, npm run app, token URLs.

2

Connecting your tools happens in a terminal

Via a /mcp command in a separate app, not in Team OS.

3

The AI brain depends on a CLI tool

Each person must install and log into it separately.

4

The daily ritual is not guided

The morning briefing is just text dumped into a chat box.

5

It does not feel finished

Sample data, chat that forgets itself, and no clear "AI is working" states.

5 The plan

Eight initiatives to close the gap

Seven move a piece of the experience from a developer's terminal into an app the team can use; the eighth is a backend upgrade under the brain. Initiative 2 is the technical linchpin.

1

One-click desktop app, no terminal ever

The single biggest unlock. It is the difference between "our team uses it" and "one person demos it," removing 100% of setup steps for non-developers.

Foundation
Today

Install Node, clone a repo, run terminal commands, copy a token URL.

→
After

Download Team OS, double-click, it opens. A native macOS app (Tauri) with runtime and engine bundled inside.

Detail
  • Ship a signed, notarized macOS build (a .dmg) so downloads open without security warnings; auto-update built in so the team never re-installs by hand.
  • The Node runtime and the AI engine are bundled inside the app — there is nothing to install, clone, or configure separately.
2

Claude as the brain, connected inside the app

Moves the last piece of developer setup into the app. This is the technical core of the plan — and there are three viable ways to wire the engine in, each with a different cost and control trade-off. We keep all three on the table rather than committing to one.

Linchpin
Today

The app launches the claude command-line tool; you connect Gmail, Slack and others in a terminal.

→
After

A one-time in-app sign-in connects the engine — via one of the three paths below — with no terminal and no separate billing setup. Plus a Connectors screen with click-to-connect and live status.

Three ways to connect the engine Verified against current Anthropic docs
  • Path A · Official Claude Agent SDK (direct). Adopt the SDK instead of spawning the raw CLI. The most capable and cleanest to build on — but it bills direct token usage per call, so cost scales with team activity. Treat it as one option, not the default: use it judiciously rather than as the sole solution.
  • Path B · AISquare collective intelligence. Route the engine through AISquare's own collective-intelligence layer instead of paying Anthropic per token directly — reusing infrastructure and spend the company already runs, which keeps token cost contained.
  • Path C · Existing Claude account via MCP auth. Connect each teammate's existing Claude subscription through a one-time MCP / OAuth sign-in that mints a long-lived token stored in the macOS Keychain. No new API keys, no new billing; leverages the seats the team already pays for.
  • Whichever path we pick, the swap is an evolution not a rewrite: the app already has a local server the UI talks to, so we replace the "spawn claude" adapter behind that same boundary, and configure connectors programmatically (each tool's OAuth runs in-app, no interactive /mcp).
  • Day-1 spike, the one thing to prove first. Confirm the chosen sign-in can be made terminal-free for non-devs, and settle the cost/control trade-off across the three paths. This is the single biggest dependency, so it goes first.
3

Guided Morning Triage, plus background rituals

The AI chief-of-staff promise: the reason someone opens the app first thing every day.

Ritual
Today

A "morning briefing" chip that pastes a command into chat.

→
After

A step-by-step flow: overnight changes, urgent inbox with draft replies to approve, who owes you a reply with one-click nudges, customer follow-ups due, delivery and blockers. Backed by an overnight scan that fills a "Needs review" tray, with desktop notifications.

Detail
  • A guided stepper walks each teammate through the ritual one stage at a time — overnight changes → urgent inbox → who owes you a reply → customer follow-ups → delivery and blockers — and every step is reviewable before it acts.
  • A background overnight scan pre-fills the "Needs review" tray and fires a desktop notification, so the briefing is ready the moment the app opens rather than being a command pasted into chat.
4

Invisible review-and-commit

The review-before-commit pattern is core to trust, but its current vocabulary is developer-only.

Trust UX
Today

Approving a change talks about branches, PRs, and "opening a pull request."

→
After

"Approve" simply means Save to the team. A clean before/after plus a "this will be visible to the team" signal, never a git term. All branch, PR and leak-check safety stays under the hood.

Detail
  • Every AI change surfaces as a plain before/after card with a single "visible to the team" signal; the git branch, pull-request, and secret/leak-check machinery runs untouched underneath, just never shown to the user.
5

Real data plus live "AI is working" states

Makes it a tool people trust with real work, not a demo.

Polish
Today

Most tabs show mock data, with limited loading, thinking, or error states.

→
After

Views render the team's real data by default, with proper streaming, "Claude is thinking," progress, empty and error states everywhere. AI work takes seconds, so the screen must never look frozen.

Detail
  • Turn off mock-by-default and wire every view to the team's real data; specify streaming, "Claude is thinking," progress, empty, and error states for each surface so the screen is never ambiguously frozen during the seconds AI work takes.
6

A real work board: kanban plus capacity

One of the most expensive recurring questions for a PM or lead, answered in seconds.

Views
Today

A basic status board.

→
After

A proper kanban board plus a capacity view that answers at a glance: who is doing what, who has the most work, who is free, who is overloaded. The PRD's "who is free by when," made visual.

Detail
  • Kanban columns for flow, plus a capacity lane that renders each person's real load — making "who is free and by when," today's most expensive recurring question, answerable in seconds instead of a day.
7

Chat that remembers

Basic reliability. The engine room cannot forget the conversation every time.

Reliability
Today

Refresh the app and your conversation disappears.

→
After

Chat history persists across refreshes and restarts, per user, locally. Pick up where you left off.

Detail
  • Persist conversations locally per user so a refresh or restart never wipes the thread; reopening the app resumes the last conversation exactly where it stopped.
8

Graphify: a knowledge graph under the brain

A backend upgrade, not a UI screen. It improves answer quality and controls token cost as the brain keeps compounding.

Backend
Today

Answers are assembled by repeatedly querying and joining flat, SQL-style stores — slower, and it burns more tokens as the context needed for an answer grows.

→
After

Graphify dynamically builds a knowledge graph from the user's actual query, so the engine traverses relationships directly instead of re-joining tables every time.

Detail
  • Query-driven graph construction means the engine pulls only the connected context an answer needs, not whole tables — cutting tokens per answer and holding cost down as the brain grows.
  • Better-connected, higher-quality answers over time; pairs with the future Gbrain graph store as the reasoning backend matures.

7 What to watch

Risks and the one thing to prove first

Biggest dependency: connecting the engine (Day-1 spike)

Three paths are viable — the Claude Agent SDK (direct, but per-token cost), AISquare's collective intelligence, or the team's existing Claude account over MCP auth. Choosing among them on cost and control, and making the first sign-in fully terminal-free for non-devs, is the item to prove before anything else. If it needs longer, it is the one thing that moves the timeline, so it is scheduled first.

Engine is evolution, not rewrite

Keep the app's existing local-server boundary and swap the CLI-spawn adapter for whichever connection path we pick behind it. Lower risk than a from-scratch engine.

Packaging

Run the engine as a bundled local service the shell talks to (avoids fragile runtime bundling), plus macOS code-signing and notarization so downloads open without scary warnings.

Connector OAuth is per-service

Gmail, Slack, Calendar and GitHub each need their own in-app OAuth flow and scopes. Build them one at a time with clear connect and failed status; silent connector failures are the classic non-dev trap.

Permissions

Pre-approve the engine's tool allow-list so non-devs never see a raw allow/deny prompt.

Token security

Store per-user tokens in the OS keychain, never in a plaintext file.
